# 新功能实现原理

本文档说明本次新增 AI 能力、问答辅助、站内引用和安全治理的实现思路。

## 一、整体架构

本次改动采用三层结构：

```text
Login_ui 前端
  -> ai_bff Python 服务
    -> Login_api Java 后端
    -> 外部大模型服务
```

前端不直接持有模型 API Key。所有模型请求统一经过 `ai_bff`，由 `ai_bff/.env` 中的 `AI_MODEL_API_KEY` 调用一个统一模型。

这样做的好处：

- API Key 不暴露到浏览器。
- 未来换模型只需要修改 BFF 配置。
- 可以在 BFF 层统一做鉴权、限流、审计、提示词治理和引用校验。
- 前端只关心业务交互，不直接依赖模型厂商协议。

## 二、AI BFF 实现原理

AI BFF 位于 `ai_bff/app/`，核心文件包括：

- `main.py`：FastAPI 入口，提供聊天、流式聊天、相似问题、AI 起草、站内搜索等接口。
- `agent.py`：智能体主逻辑，负责上下文收集、提示词组装、模型调用和结果后处理。
- `model_client.py`：OpenAI-compatible 模型调用封装。
- `login_api_client.py`：调用 Java 后端接口，获取笔记、问答、搜索结果等站内数据。
- `schemas.py`：请求和响应结构定义，并限制输入长度。
- `auth.py`：JWT 校验。
- `site_retrieval.py`：站内检索、关键词提取、引用构造和存在性校验。
- `reference_markers.py`：把模型输出中的站内 marker 转换为可点击链接。
- `copy.py`：统一用户可见文案。

## 三、聊天和站内检索流程

用户在前端 AI 面板输入问题后：

1. `Login_ui` 将用户输入、当前页面、当前用户、当前资源等上下文发送给 `ai_bff`。
2. `ai_bff` 校验 JWT，确认请求来自已登录用户。
3. `agent.py` 根据问题判断是否需要站内检索。
4. 如果需要检索，`site_retrieval.py` 会提取关键词。
5. `login_api_client.py` 调用 `Login_api` 搜索笔记或问答。
6. 检索命中后，BFF 会再次校验笔记是否真实存在。
7. BFF 把真实命中内容作为上下文事实交给模型。
8. 模型输出后，BFF 清理外部 URL，解析站内 marker，生成结构化 citations 和 answerLinks。
9. 前端渲染 Markdown，并把站内链接拦截为应用内跳转。

## 四、站内引用如何避免造假

模型不能直接决定 `noteId` 或 `questionId` 是否有效。

本次实现中，引用必须来自 BFF 同一轮检索得到的真实结果：

- 笔记引用必须包含真实 `noteId`。
- 问答引用必须包含真实 `questionId`。
- 笔记引用会通过 Login_api 读取接口二次校验。
- marker 中不存在于 citations 的 ID 会被降级为普通文本。
- 空检索时返回统一无结果文案，不展示假引用。

引用结构示例：

```json
{
  "type": "note",
  "title": "新建笔记",
  "noteId": 1,
  "route": {
    "tab": "note-detail",
    "noteId": 1
  }
}
```

前端只对通过 `isRouteableCitation` 校验的引用允许跳转。

## 五、Markdown 和内联链接渲染

前端新增 `Login_ui/src/utils/aiMarkdown.js`：

- 使用 `markdown-it` 渲染 AI 回答。
- 使用 `dompurify` 清洗 HTML。
- 只允许 `folio://note/...` 和 `folio://question/...` 作为站内链接。

AI 面板中点击 Markdown 链接时，不直接让浏览器跳转，而是：

1. 解析 `folio://` 链接。
2. 在当前消息的 citations 中寻找对应引用。
3. 构造应用内导航目标。
4. 通知 `MainView.vue` 切换到笔记详情或问答详情。

这样可以避免模型生成任意外部链接造成安全风险。

## 六、问答区 AI 起草原理

问答区新增两个能力：

### 1. 相似问题检测

用户打开提问弹窗并输入标题或正文时，前端会 500ms debounce 后调用：

```text
POST /api/v1/agent/similar-questions
```

BFF 调用 Login_api 的问答搜索接口，只返回真实存在的问题。没有结果时返回：

```json
{
  "items": []
}
```

不会再返回占位问题或伪造 ID。

### 2. AI 起草问题

用户输入关键词后，前端调用：

```text
POST /api/v1/agent/draft-question
```

BFF 要求模型返回 JSON：

```json
{
  "title": "问题标题",
  "content": "问题正文",
  "tags": ["标签1", "标签2"]
}
```

如果模型返回不是合法 JSON，BFF 会用本地 fallback 生成一个安全草稿。前端展示预览，用户必须点击采纳后才会写入发帖表单。

## 七、写操作和安全边界

当前 AI 主要做建议和辅助，不直接替用户保存内容。

关键边界：

- 浏览态不允许直接改写正文。
- 编辑态改写也需要用户确认后提交。
- AI 起草问题不会自动发布。
- 模型 API Key 只在 BFF 读取。
- BFF 需要 JWT 才提供完整能力。
- 超长输入会被拒绝或截断，避免异常请求影响服务。

## 八、后端接口和迁移原理

### 1. 站内目录接口

`SearchController` 新增：

```text
GET /api/v1/search/notes/catalog
```

用于 AI broad-list 检索。当用户问“有哪些笔记”“推荐一些文章”时，BFF 可以先拿站内公开笔记目录，再做引用构造。

### 2. Flyway 迁移清理

Flyway 要求迁移文件符合 `V数字__描述.sql` 格式。

旧文件：

```text
V4_create_note_stat_table.sql
V5_create_user_follow_table.sql
```

缺少双下划线，开启 `validate-migration-naming` 后会被判定为非法。

因此迁移文件被调整为规范命名，并把相关表拆成：

- `V5__create_user_follow_table.sql`
- `V7__create_note_stats.sql`
- `V8__create_note_stats_compensation.sql`
- `V10__create_user_favorite_note_table.sql`

## 九、全站一个模型的当前策略

当前推荐策略是全站共用一个模型配置：

```env
AI_MODEL_PROVIDER=openai_compatible
AI_MODEL_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
AI_MODEL_NAME=qwen3-vl-plus
AI_MODEL_API_KEY=真实key
```

所有 AI 能力都走同一条 BFF 调用链：

```text
前端功能 -> ai_bff -> 统一模型客户端 -> 模型服务
```

后续如果需要分模型，可以在 BFF 层扩展路由策略，例如：

- 聊天使用通用模型。
- 起草使用文本模型。
- 文档理解使用多模态模型。
- 高风险操作使用更严格的安全 prompt。

前端不需要为这些变化改动密钥管理方式。
